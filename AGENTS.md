# AGENTS.md

本文件用于减少常见 LLM 编码错误，并约束代理在本仓库中的协作方式。若子目录存在更具体的 `AGENTS.md`，以更近的文件为准；若用户当前明确要求更具体，优先满足用户要求。

## 1. 中文协作优先

- 代理与用户的所有交互必须使用中文，包括计划、进度更新、最终回复、问题澄清和变更说明。
- MR/PR 相关协作文本必须使用中文，包括标题、描述、评论、代码审查回复、议题回复、提交说明和发布/变更总结。
- 自动生成的提交信息必须使用中文，格式为 `<类型>: <简要描述>`，例如 `修复: 处理配置读取失败`。
- 允许保留必要原文：代码标识符、命令、文件路径、协议字段、第三方 API 名称、英文错误原文和引用内容。
- 用户可见的产品文本仍必须走 i18n 机制，不要为了中文协作要求而在代码里硬编码单一语言字符串。

## 2. 先明确目标

- 开始修改、运行命令或调查问题前，先给出简要计划，说明要做什么以及如何验证。
- 简单任务使用 1 到 2 步短计划即可；复杂任务需要拆分阶段，并为每阶段定义可验证结果。
- 不确定时先说明假设和歧义；高影响问题必须先澄清，不要默默选择。
- 如果发现更简单、更低风险的方案，明确指出并优先采用。
- 任务范围变化时及时更新计划，不要沿用过期计划。

## 3. 简单且精准

- 只实现当前请求需要的内容，不添加未要求的功能、配置项或抽象。
- 匹配现有代码风格和模块边界，不顺手重构、改格式或清理无关代码。
- 每一行改动都应能追溯到用户请求或本次修改引入的必要清理。
- 如果本次修改造成无用导入、变量、函数或测试辅助代码，应一并移除。
- 发现预先存在的无关问题时，只在回复中说明，不要擅自修复。

## 4. 实现质量

- 优先使用仓库已有模式、类型、工具函数和成熟开源库，不重复造轮子。
- 对 JSON、YAML、TOML、XML 等结构化数据必须使用对应解析库，禁止用正则或脆弱字符串匹配解析。
- 保持类型安全，避免魔法字符串、含糊布尔参数和难以理解的位置参数。
- 函数保持单一职责；新抽象只有在减少真实复杂度或匹配既有模式时才添加。
- 面向外部输入、文件系统、网络、并发和时间的代码必须处理可预见失败，并给出包含上下文的错误信息。
- 防御性处理应聚焦真实风险，不为不可能或无需求的场景堆砌复杂逻辑。

## 5. 国际化

- 所有用户可见文本必须支持国际化，包括错误信息、提示、标签、状态和帮助文本。
- 新增或变更用户可见文本时，同步更新中文和英文语言资源。
- 日期、数字、货币等格式化内容使用本地化 API。
- 避免拼接自然语言句子；不同语言的语序、复数和语法可能不同。
- 验证时应确认切换语言后界面文本完整、准确，没有遗漏硬编码文本。

## 6. 验证与交付

- 缺陷修复优先添加能复现问题的测试，再让测试通过。
- 新功能和重构需要覆盖主要行为、边界和失败场景；测试范围随风险和影响面扩大。
- 运行与改动最相关的格式化、lint、测试或构建命令；无法运行时说明原因和剩余风险。
- 长耗时任务需要持续给出中文进度，让用户知道当前阶段、原因和剩余工作。
- 最终回复用中文说明完成内容、验证结果、未完成项和需要用户注意的风险。

## 7. Git 与工作区

- 不要回退、覆盖或格式化用户已有改动，除非用户明确要求。
- 工作区有无关改动时保持隔离，只处理本次任务相关文件。
- 禁止使用破坏性 Git 命令，如 `git reset --hard` 或 `git checkout --`，除非用户明确要求并确认目标。
- 提交前确认 diff 只包含本次请求需要的改动。

这些准则有效的标志是：改动更小、歧义更早暴露、验证更明确，MR/PR 与日常协作文本都保持中文且易于审查。
