# CLAUDE.md

减少常见 LLM 编码错误的行为准则。根据需要与项目特定说明合并。

**权衡：** 这些准则倾向于谨慎而非速度。对于简单任务，请自行判断。

## 1. 先思考再编码

**不要假设。不要隐藏困惑。明确权衡取舍。**

在实现之前：
- 明确陈述你的假设。如果不确定，请询问。
- 如果存在多种解释，请列出它们——不要默默选择。
- 如果存在更简单的方法，请说出来。必要时提出异议。
- 如果有不清楚的地方，停下来。指出困惑所在。询问。

## 2. 简单优先

**解决问题的最少代码。不做推测性开发。**

- 不添加未被要求的功能。
- 不为一次性代码创建抽象。
- 不添加未被要求的"灵活性"或"可配置性"。
- 不为不可能的场景添加错误处理。
- 如果你写了 200 行而 50 行就够了，重写它。

问自己："高级工程师会说这太复杂了吗？"如果是，简化它。

## 3. 精准修改

**只改动必须改的。只清理自己造成的混乱。**

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有问题的东西。
- 匹配现有风格，即使你会用不同的方式。
- 如果发现无关的死代码，提一下——不要删除它。

当你的更改产生孤立代码时：
- 移除因你的更改而变得无用的导入/变量/函数。
- 除非被要求，不要移除预先存在的死代码。

检验标准：每一行更改都应该直接追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后使其通过"
- "修复 bug" → "编写重现 bug 的测试，然后使其通过"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，陈述简要计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

强有力的成功标准让你能够独立循环。弱标准（"让它工作"）需要不断澄清。

## 5. 计划与并行执行

**每次执行任务都要先给出计划。可并行时主动拆分。**

- 开始修改、运行命令或调查问题前，先给出简要 plan。
- plan 至少包含：要做什么、如何验证。
- 简单任务可以使用 1-2 步短计划，但不能省略。
- 任务过程中如果范围变化，及时更新 plan，而不是继续执行旧计划。
- 当存在互不依赖、可并发的调查或实现子任务时，可以指挥多个 subagent 并行处理。
- 使用 subagent 前要明确每个 subagent 的职责、输入、预期输出和边界。
- 汇总 subagent 结果时，由主 agent 做最终判断；不要直接拼接结论。

## 6. Git 提交规范

**自动生成的 commit message 必须使用中文。**

- 提交信息使用简洁的中文描述更改内容。
- 格式：`<类型>: <简要描述>`
- 常用类型：`功能`、`修复`、`重构`、`文档`、`样式`、`测试`、`构建`
- 示例：`功能: 添加用户登录验证`、`修复: 解决空指针异常`

## 7. 国际化要求

**所有用户可见的文本必须支持国际化。**

开发时必须遵守：
- 不在代码中硬编码用户可见的字符串（错误信息、提示、标签等）。
- 所有面向用户的文本必须通过 i18n 机制提取，支持多语言切换。
- 新增功能时，同步更新中文（zh）和英文（en）语言文件。
- 日期、数字、货币等格式化内容使用本地化 API。
- 避免字符串拼接构建句子——不同语言语序不同。
- 复数形式、性别变化等需考虑目标语言的语法规则。

检验标准：切换语言后，所有用户界面文本应正确显示，无硬编码字符串遗漏。

## 8. 原则

**编写高质量、可维护、可扩展的代码。**

1. **模块化设计**：各功能模块独立管理资源（包括国际化），避免耦合。
2. **类型安全优先**：使用接口、枚举、常量约束，避免魔法字符串和硬编码比较。
3. **按需实现**：只做当前必要的功能，拒绝推测性设计和过早优化。
4. **善用开源生态**：TUI、CLI、格式化输出等通用场景，优先选择成熟的开源库。
5. **可读性至上**：代码结构清晰，命名语义化，便于他人理解和后续维护。
6. **正确解析结构化数据**：JSON、YAML、TOML、XML 等必须使用对应解析库，禁止正则匹配。
7. **用户交互友好**：所有用户交互界面必须支持国际化，提供清晰的提示和反馈。
8. **错误处理**：所有可能出错的地方都必须有错误处理机制，避免程序崩溃。
9. **函数单一职责**：每个函数只负责一个明确的任务，避免复杂逻辑。

## 9. 健壮性编码

**预见边界情况，防御性编程，优雅降级。**

输入验证：
- 永远不要信任外部输入——校验类型、范围、格式。
- 处理 `null`、`undefined`、空字符串、空数组、空对象。
- 字符串输入考虑：超长文本、特殊字符、Unicode、前后空白、换行符。
- 数值输入考虑：零、负数、小数、极大值、极小值、NaN、Infinity。
- 集合输入考虑：空集合、单元素、超大集合、重复元素。

边界条件清单：
- **数组/列表**：空、单元素、首尾元素、越界索引。
- **字符串**：空串、单字符、多字节字符（emoji、中文）、仅空白。
- **数值**：0、1、-1、最大值、最小值、浮点精度。
- **时间**：跨时区、夏令时、闰年、月末、年末、时间戳溢出。
- **文件**：不存在、无权限、被占用、路径过长、特殊字符路径。
- **网络**：超时、断连、重试、部分响应、非预期状态码。
- **并发**：竞态条件、死锁、资源争用。

错误处理原则：
- 区分可恢复错误和致命错误，分别处理。
- 错误信息包含上下文：发生了什么、在哪里、为什么、如何修复。
- 不要吞掉异常——至少记录日志。
- 资源（文件、连接、锁）必须在 finally/defer 中释放。
- 考虑部分失败：批量操作中单项失败时的回滚或跳过策略。

防御性编程：
- 对外部依赖（API、数据库、文件系统）的返回值做校验。
- 使用断言验证内部不变量，快速暴露逻辑错误。
- 配置合理的超时，避免无限等待。
- 限制递归深度，防止栈溢出。
- 限制循环次数，防止死循环。

检验标准：问自己"如果输入是 X 会怎样？"——X 包括 null、空、极值、畸形数据、并发访问。

## 10. 交互体验

**让程序运行过程可见、可理解、可信赖。**

1. **空闲厌恶感** — 持续输出进度，证明程序存活。即使是大模型推理，也要用 spinner、逐字输出或状态更新，避免用户面对静止屏幕产生焦虑。

2. **运营透明度** — 明示当前在做什么、原因、进度。例如：`[3/12] 扫描 ~/.codex/skills … 发现 7 个 skill`，而非静默等待后抛出结果。

3. **目标梯度效应** — 越接近完成越要体现进展，驱动用户等待。进度条在最后 20% 不应停滞；阶段性完成时给出 ✓ 确认，让用户感知剩余工作量在减少。

---

**这些准则有效的标志是：** diff 中不必要的更改更少，因过度复杂化而重写的次数更少，澄清性问题在实现之前提出而非在犯错之后。
